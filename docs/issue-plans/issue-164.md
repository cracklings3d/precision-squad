# Summary

The `graphify` skill was run on this repository and produced `graphify-out/` containing a knowledge graph (479 nodes, 606 edges, 75 communities) and an HTML visualization. This issue formally accepts the output, adds it to `.gitignore` to prevent accidental commits, documents a refresh policy, produces a triage report of the 235 isolated nodes, and records a decision on HTML publishing.

# Problem

The `graphify-out/` directory was generated but is untracked. Without a `.gitignore` entry, it could be accidentally committed. Additionally, the project has no policy for when or how to regenerate the graph, and the 235 isolated nodes in the graph represent potential documentation gaps that should be reviewed without being fixed as part of this PR.

# Acceptance Criteria

- [ ] `graphify-out/` is present in `.gitignore` and remains uncommitted
- [ ] A "Graph Refresh Policy" section exists in `CONTEXT.md`
- [ ] `docs/graphify-gap-report.md` exists, listing isolated nodes grouped by likely cause with a "Recommended follow-up issues" section
- [ ] The issue body records the decision on HTML publishing (default: gitignore) with rationale
- [ ] No source code is refactored and no graph edges are added in this PR

# In Scope

- Add `graphify-out/` to `.gitignore`
- Extract the 235 isolated-node list from `graphify-out/GRAPH_REPORT.md` Section "Knowledge Gaps" and group them by likely cause into `docs/graphify-gap-report.md`
- Add a "Graph Refresh Policy" section to an appropriate doc (see Proposed Approach)
- Record the HTML publishing decision in the issue body

# Out Of Scope

- Refactoring source code to fix isolated-node gaps
- Adding edges between knowledge-graph nodes
- Committing `graphify-out/` artifacts to the repo
- Creating follow-up issues (the triage report recommends them; this PR does not file them)

# Constraints

- Per CONTEXT.md "one issue per PR" rule: this PR must not refactor source or add edges
- The graph can be regenerated at any time via `/graphify` — it is not a build artifact

# Proposed Approach

## Step 1: Add `graphify-out/` to `.gitignore`

Append `graphify-out/` to `.gitignore`. Confirm no other generated-output directories are missing.

## Step 2: Document Refresh Policy

The graph is regenerated on-demand via `/graphify`. Add a "Graph Refresh Policy" section to `CONTEXT.md` adjacent to existing tooling/skill conventions. The section reads:

> **Graph Refresh Policy** — The knowledge graph is generated by the `graphify` skill (`/graphify`). Run it on demand after major refactors or architectural changes. The output under `graphify-out/` is not committed; see `.gitignore`.

## Step 3: Produce `docs/graphify-gap-report.md`

Parse `graphify-out/GRAPH_REPORT.md` Section "Knowledge Gaps":
- 235 isolated nodes with ≤1 connection are listed
- Group them by likely cause: test fixtures, deprecated modules, undocumented CLI handlers, bootstrap utilities, archived docs, etc.
- End with a "Recommended follow-up issues" section: a flat list of follow-up issue titles (not filed yet — filed separately after this PR merges)

**The gap report is committed** (`docs/graphify-gap-report.md`) as a snapshot with lasting value for backlog grooming. It is not a temporary working document.

## Step 4: Record HTML Publishing Decision

Default: **do not commit `graphify-out/`**. Rationale: graph is regeneratable on-demand. If external consumers need HTML hosting, that decision requires explicit justification in a separate issue.

Update the issue body with:
> **HTML Publishing Decision:** `graph.html` is excluded from the repo by `.gitignore`. The graph can be regenerated and served externally. If a use case for committed HTML emerges, a separate issue will document the rationale for the deviation.

# Risks

- **Risk:** The triage report could grow large (235 items) and become hard to maintain.
  **Mitigation:** The report is a snapshot at point-in-time; future graphify runs produce a fresh report. The report is guidance, not a contract.
- **Risk:** The refresh policy section could conflict with existing documentation conventions.
  **Mitigation:** Place it adjacent to similar tooling-policy entries rather than creating a new standalone doc.
