# Precision Squad Operator Skill

Use this document when an AI agent or human operator is driving normal development work with GitHub and `precision-squad` together.

## Purpose

This skill teaches an operator how to:

- choose the right workflow
- run the CLI correctly
- interpret outcomes
- follow docs-remediation issues without creating duplicate work
- know when the control plane is behaving correctly versus when the repair agent is weak

## Primary Workflows

There are two top-level commands:

1. `repair issue`
2. `publish run`

### `repair issue`

Use this when you have a GitHub issue and want `precision-squad` to:

- inspect the repo docs
- attempt repair if allowed
- evaluate the result
- optionally publish the outcome

Command:

```bash
python -m precision_squad.cli repair issue owner/repo#number --repo-path <local-repo> --publish
```

### `publish run`

Use this when a run already exists locally and you want to publish its stored result without rerunning repair.

Command:

```bash
python -m precision_squad.cli publish run <run-id>
```

## Normal Decision Tree

When an operator is given a daily development task, use this sequence.

1. Start with an existing GitHub issue.
2. Run `repair issue ... --publish` against the local checkout.
3. Inspect the publish result.

Interpretation:

- `Publish Plan: draft_pr`
  - the run is approved or provisional enough to publish a PR
- `Publish Plan: follow_up_issue`
  - the original issue is blocked by a separate repo-level docs/setup problem
- `Publish Plan: issue_comment`
  - the issue itself should receive blocked feedback rather than creating new GitHub work

## Feature Issue Versus Docs-Remediation Issue

### Feature Issue

A normal feature or bug issue should usually do one of these:

- repair and publish a PR
- fail on repo-level docs blockers and create a follow-up docs-remediation issue

### Docs-Remediation Issue

A docs-remediation issue is a special issue generated by `precision-squad`.

It is identified by hidden markers in the issue body:

- `<!-- precision-squad:docs-remediation -->`
- `<!-- precision-squad:target-findings:... -->`
- `<!-- precision-squad:baseline-findings:... -->`

When the operator sees these markers, treat the issue as a remediation issue, not a normal product issue.

Important behavior:

- it should not create another remediation issue for itself
- it should rerun against the same target findings
- it should post blocked feedback back onto the same issue if target findings remain

## Meaning Of Findings

Docs-remediation issues track structured findings, not just prose summaries.

A finding contains:

- `rule_id`
- `source_path`
- `section_key`
- `subject_key`

Example:

```json
{
  "rule_id": "docs_setup_prerequisite_manual_only",
  "source_path": "readme.md",
  "section_key": "windows-system-dependencies",
  "subject_key": "gtk3-runtime"
}
```

Interpretation:

- `target_findings`
  - what this remediation issue is supposed to eliminate
- `baseline_findings`
  - blockers already known elsewhere and not in scope for this issue

Success rule for remediation issues:

- all target findings must disappear

Blocked-but-acceptable rule:

- baseline findings may remain if target findings are cleared

Failure rule:

- target findings remain, or new untracked findings appear

## What To Do On Reruns

If a docs-remediation issue stays blocked, rerun the same issue again:

```bash
python -m precision_squad.cli repair issue owner/repo#number --repo-path <local-repo> --publish
```

Why rerun:

- the latest blocked comment is pulled from GitHub
- that comment is written into `issue.md`
- the repair agent sees the latest target findings and prior failure summary

This is the intended repair-feedback loop.

## How To Judge The System

The operator should distinguish between:

### Control Plane Success

The system is behaving correctly when it:

- keeps the original issue focused
- creates a remediation issue for unrelated repo blockers
- avoids duplicate remediation issues
- reuses existing remediation issues when findings match
- reruns remediation issues in place
- reports remaining target findings precisely

### Repair Agent Failure

The system may still be functioning even when the repair agent is weak.

Examples:

- the issue is rerun correctly
- prior blocked feedback appears in `issue.md`
- the patch changes each time
- but the same target findings still remain

That means the loop is working, but the agent is not converging.

## When To Stop Rerunning Blindly

Do not rerun forever.

Pause and reassess when:

- the same target findings remain across multiple reruns
- the agent keeps rewriting prose inside the same invalid solution family
- the docs policy may be stricter than the underlying dependency can honestly satisfy

At that point the operator should decide whether:

1. the prompt needs tightening
2. the policy needs adjustment
3. the underlying dependency should be replaced

## Example Daily Task Patterns

### Normal feature request

1. Find or create the GitHub issue.
2. Run `repair issue ... --publish`.
3. If it creates a PR, review the PR.
4. If it creates a remediation issue, switch to that issue.

### Docs blocker follow-up

1. Run the remediation issue with `repair issue ... --publish`.
2. Read the blocked comment or PR result.
3. If still blocked, rerun the same issue only if the latest feedback meaningfully sharpens the target.

### Stored-run publication

1. Identify the run ID.
2. Run `publish run <run-id>`.
3. Let post-publish review execute if applicable.

## Current Limits

An operator should know these limits:

- issue creation is still manual/operator-driven
- repair quality may be weaker than control-plane quality
- some Windows runtime dependencies may be hard to express under the current strict docs policy
- repeated reruns do not guarantee convergence

## Recommended Operator Style

- use GitHub issues as the task boundary
- run the CLI from a real local checkout of the target repo
- publish outcomes so future reruns inherit GitHub feedback
- prefer rerunning the same remediation issue over creating new manual duplicates
- treat precise target findings as the real truth, not the high-level prose summary
